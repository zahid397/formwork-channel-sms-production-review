## Context — read this first

You are given `formwork-channel-sms`, a real Spring Boot module from a
multi-tenant platform. It sends SMS through five providers (Twilio, Vonage,
MessageBird, BudgetSMS, AWS SNS) behind a common `SmsGateway` interface, and
records per-tenant cost.

Two things you should know:

1. **Most of this code was generated by an AI coding agent.** It compiles. The
   tests pass. It looks professional. It also contains bugs that would cost real
   money, leak personal data, and fail in production.

2. **The engineer we hire will be the only backend engineer on this project.**
   Nobody will review your pull requests. Nobody will catch what you miss. The
   standards you set are the standards that ship.

So this assignment is not "can you write Spring Boot." It is: **can you look at
plausible-looking AI output and find what's wrong with it?** That is the single
skill this role lives or dies on.

We use Claude Code every day and expect you to. We are not testing whether you
can avoid AI. We are testing whether you are the engineer who *reviews* it, or
the one who *ships* it unread.

## Part 1 — Review (the main event)

Read the module. Write `REVIEW.md` containing **at least 8 findings**.

For each finding:

- **Severity** — Critical / High / Medium / Low, and say what a user or the
  business actually loses.
- **Location** — `file:line`.
- **What's wrong** — the mechanism, not a label. "Not thread-safe" is a label.
  "Two concurrent calls can interleave between the `get` and the `put`, so one
  update is lost" is a mechanism.
- **How you'd fix it.**

At least **two** findings must be defects that would cause a production
incident — money lost, data leaked, or requests failing — not style issues.

We have our own list. We are interested in what you find, what you rank highest,
and what you find that we didn't.

> Look hardest at: the AWS SNS gateway, the cost pipeline end to end, and the
> test suite. Ask yourself of each test: *if I broke the code this test covers,
> would this test fail?*

## Part 2 — Fix, and prove it

Pick the **three highest-severity findings** from your review and fix them.

For each fix, commit a test that **fails on the original code and passes on
yours**. Make this visible: either a commit where the test is added and red, or
a note in the PR/README pointing at the commit SHA. A fix without a failing test
first is a claim, not evidence.

## Part 3 — Make it work

The module has holes where features were clearly intended and never wired up.
Close them:

1. **Tenant-aware provider selection.** `SmsMessage` carries a `tenantId` that
   currently changes nothing. A tenant should be able to be on a different
   provider than the global default, and one tenant's configuration must never
   affect another's.

2. **Cost recording must actually happen.** Find out why it doesn't today, then
   make the send path record cost correctly — including for messages that span
   more than one segment.

3. **Retry and failover.** `SmsChannelProperties.RetryProperties` exists and
   nothing uses it. Implement retry with a sensible backoff, and failover to a
   secondary provider. Be deliberate about which failures are worth retrying —
   retrying the wrong one is worse than not retrying at all.

4. **One honest integration test.** Add a test that exercises a gateway over
   real HTTP against a stub server, asserting on the bytes actually sent — the
   URL, headers, and body. (Look at what the existing `*WireMockTest` files do
   before you decide how much of the existing suite you trust.)

You do not need to implement all four perfectly. We would rather see two done
properly, with the other two scoped and explained in your README, than four done
carelessly. Say what you cut and why.

## Part 4 — Own it

You are the only engineer here. Show us what that looks like:

- **`README.md`** — how to build and run, what you changed, what you'd do next
  with another week, and what you deliberately did not do.
- **One ADR** (`docs/adr/0001-*.md`, one page) for the most consequential design
  decision you made. State the alternatives you rejected and why.
- **CI** — a GitHub Actions workflow that builds, tests, and fails the build if
  coverage drops below a threshold you choose and justify.
- **`AI-USAGE.md`** — see below.

## `AI-USAGE.md` — required

Use Claude Code or any AI tool as much as you like. Then tell us the truth about
it, in a file named `AI-USAGE.md`:

- Which parts did you generate, and roughly how?
- **Where was the AI wrong?** Show us at least one concrete case: what it
  produced, why it was wrong, and what you replaced it with.
- What did you write yourself because you didn't trust the agent with it?

There is no penalty for heavy AI use. There is a penalty for not knowing what
your own code does. A candidate who says "the agent generated the retry logic, I
rewrote the backoff because its jitter was applied after the cap, which made the
cap meaningless" tells us more than a candidate who wrote it all by hand.

**We will ask you to defend any line in your submission during the interview.**
Submit nothing you cannot explain.

## How we score

| Weight | What we look at |
|---|---|
| **40%** | Part 1 — depth of review. Did you find the real bugs, or the cosmetic ones? |
| **25%** | Part 2 — are the fixes correct, and did you prove it with tests? |
| **20%** | Part 3 — engineering judgment: what you built, what you cut, and why |
| **15%** | Part 4 — could a solo owner maintain this? Docs, ADR, CI, honesty |

What earns nothing: a rewrite from scratch, 100% coverage of trivial getters, or
a `REVIEW.md` that lists twenty formatting nits and misses the money bug.
